Complete REST API Design

Chapter 1

REST began as a scalability answer, not a routing style

The source starts with web history because REST only makes sense once its architectural purpose is visible.

From the early web to Roy Fielding's constraints

Tim Berners-Lee introduced the early web stack with URI, HTTP, HTML, the first web server, and the first browser. As the web scaled beyond its original research setting, the challenge stopped being simple connectivity and became large-scale system behavior.

Roy Fielding's PhD work in 2000 named REST as an architectural style for networked systems. The point was not to invent a fashionable API acronym; it was to define constraints that preserve scalability, maintainability, and interoperability as systems grow.

Representational state transfer is about moving resource state through HTTP

The source breaks the term into three ideas. A resource can be represented in formats such as JSON, XML, or HTML. State refers to the current attributes of that resource. Transfer is the movement of that representation between client and server.

That framing matters because a REST API is not primarily a list of verbs. It is a contract for transferring representations of resources through a uniform interface.

Clarified note

  • Representation: the format a client receives or sends, such as JSON.
  • State: the current attributes of a resource, such as a cart's items or a task's status.
  • Transfer: the HTTP exchange that moves that representation across the network.

Chapter 2

REST constraints make the interface predictable at scale

The source lists the six REST constraints because they explain why RESTful APIs feel standardized across teams and systems.

The six constraints in practical backend language

Client-server separation keeps interface concerns on the client and data plus business logic on the server. Statelessness means each request carries what the server needs, which improves reliability and load-balancing flexibility. Cacheability allows responses to declare reuse so repeated work can be avoided.

Uniform interface standardizes the contract itself. Layered systems allow proxies, load balancers, and gateways to exist without breaking the API mental model. Code on demand is optional and appears far less often in everyday backend practice.

Client-serverSeparate UI from data and business logicBoth sides can evolve more independently
Uniform interfaceUse consistent resource-driven communicationConsumers learn one predictable model
Layered systemAllow intermediaries between client and originImproves flexibility, scale, and security
CacheableMark responses that can be reusedReduces latency and load

Code on demandOptional executable code from server to clientRare in everyday API design

Uniform interface is the constraint most API designers feel every day

The source emphasizes that REST APIs feel intuitive when resource identification, message structure, and method semantics line up consistently. This is why naming, status codes, response shapes, and route patterns matter so much.

A team can break REST without noticing by mixing route styles, inventing ad hoc verbs, or treating similar resources differently. Consistency is what turns the abstract constraint into a usable developer experience.

Chapter 3

Good REST design begins with resources, URLs, and hierarchy

Once the architectural frame is clear, the source moves into route design decisions that backend engineers make directly.

A URL is a resource address plus optional query controls

The source breaks a URL into scheme, authority, path, query parameters, and fragment, then notes that fragments are not especially relevant for typical APIs. The important design work happens in the path and in query parameters used for filtering, sorting, and pagination.

That is why a route like /organizations/{orgId}/projects/{projectId} communicates more than location. It expresses resource hierarchy and gives the client a stable mental model of how the system is organized.

Plural nouns, lowercase slugs, and stable hierarchy reduce ambiguity

The source recommends plural resource names, lowercase letters, and hyphenated slugs instead of spaces or underscores. Those conventions keep URLs consistent across environments and avoid awkward singular-versus-plural debates for each endpoint.

Resource discovery often starts from UI wireframes or product nouns: users, projects, tasks, organizations, books. Once the nouns are stable, the route surface becomes much easier to design.

Chapter 4

HTTP methods and CRUD conventions should say exactly what the endpoint does

The source moves from naming to method semantics, because a good route shape still needs a reliable verb model.

Method semantics and idempotency shape backend behavior

GET retrieves resources and is safe plus idempotent. POST creates new resources or triggers custom actions and is the common non-idempotent method. PUT replaces a full resource, PATCH partially updates it, and DELETE removes it while remaining idempotent from the server-state perspective.

Idempotency matters because retrying a request should not accidentally create duplicate state when the method promises stability. That is one reason REST APIs feel more dependable when method semantics are respected rather than improvised.

MethodTypical useIdempotentNotes

CRUD routes are the baseline, not the whole API

The source gives the standard surface for a resource: GET /resources, POST /resources, GET /resources/{id}, PATCH /resources/{id}, and DELETE /resources/{id}. This pattern is intentionally repetitive because repetition here improves clarity for clients and maintainers.

Not every operation fits plain CRUD. When the action is domain-specific, such as archiving an organization or publishing a draft, the source recommends POST to a descriptive action endpoint rather than distorting GET or PATCH beyond recognition.

Chapter 5

Pagination, status codes, and documentation turn a correct API into a usable one

The final source sections focus on what makes an API pleasant and dependable for consumers after the basic route surface exists.

List endpoints need predictable controls for large result sets

The source recommends pagination, sorting, and filtering as first-class parts of list APIs. Query parameters such as page, limit, sortBy, sortOrder, or field-based filters keep collection routes flexible without turning the path into an argument dump.

Server-side safe defaults matter here. A backend should not require every consumer to remember pagination rules or sorting order just to avoid accidental heavy queries.

Status codes and interface documentation complete the contract

The source uses familiar status conventions: 200 for successful reads, 201 for creation, 204 for successful deletion with no response body, and 404 when a resource does not exist. These codes work because they let the transport layer communicate meaning without custom interpretation.

It also recommends designing before coding and publishing interactive documentation such as Swagger or OpenAPI. That step reduces team confusion and makes the API easier to consume without reverse-engineering endpoints from trial and error.

Overall takeaway

What to remember

A strong REST API starts with resource nouns, keeps URL design and method semantics consistent, uses status codes honestly, and adds usability features such as pagination and documentation. The source's deeper point is that predictability is itself a backend feature.
