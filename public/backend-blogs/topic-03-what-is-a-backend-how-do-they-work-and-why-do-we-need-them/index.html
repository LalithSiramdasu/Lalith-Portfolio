Backend Series / Topic 3

What Is a Backend, How Does It Work, and Why Do We Need It?

Topic 3 moves from abstract backend thinking to a concrete system view. It explains what a backend traditionally is, how requests reach it across infrastructure layers, and why frontend environments cannot safely or efficiently replace backend logic.

Chapter 1

The backend as a server-side system

The source note begins with the traditional definition of a backend. It frames the backend as a computer that listens for incoming requests on open ports and serves or accepts data over the network.

Understanding the backend: traditional definition and role

In the note, the backend is introduced as a server that listens for HTTP, WebSocket, gRPC, and similar request types on Internet-accessible ports such as 80 or 443. That server delivers content back to clients and accepts incoming data based on the request type and application behavior.

  • It may serve static files such as HTML, JavaScript, and images.
  • It may return structured data such as JSON.
  • It may receive and process client-submitted data.

Chapter 2

How a request reaches the backend

The source note uses an AWS EC2 example to make the backend request path concrete. This flow moves through domain resolution, infrastructure filtering, reverse proxy handling, and finally the application process itself.

Holistic view of backend flow

The example backend is deployed on AWS EC2, and the note uses browser network tools to trace the full request-response path. The point is not just to show a response, but to make the journey visible from the browser all the way to the server.

DNS role in routing requests

The request begins with the domain name. DNS resolution then maps that domain or subdomain to a destination so the browser knows where the request should go.

  • A records point a domain or subdomain to an IP address.
  • CNAME records point one domain or subdomain to another.
  • The note uses an example where a backend subdomain resolves to an EC2 public IP.

Mapping DNS to AWS and passing the security boundary

Once DNS resolves the target IP, the request reaches the EC2 instance. Before it touches server software, it passes through the AWS security layer enforced by the Security Group.

Security Group ports named in the source

  • 22 for SSH
  • 80 for HTTP
  • 443 for HTTPS

Requests sent to ports outside those allowed rules are blocked before they ever reach the application stack.

Reverse proxy with NGINX and local application routing

NGINX is described as a reverse proxy sitting in front of the application. It listens on public ports, handles redirection from HTTP to HTTPS, and routes requests for a domain to a local backend process running on the same machine.

  • The example backend process runs on localhost port 3001.
  • NGINX provides centralized request handling and cleaner service management.
  • Process management for the Node.js service is shown through PM2.

End-to-end request flow summary

Request path in the source note

  1. Browser sends the request.
  2. DNS resolves the domain to an IP address.
  3. AWS Security Group filters access.
  4. NGINX receives and redirects or routes the request.
  5. The Node.js backend on localhost handles the request.

Chapter 3

Why backend systems are needed

After showing what a backend is and how requests reach it, the note explains why modern products depend on centralized backend behavior rather than just client-side code.

Practical example: a like on Instagram

The source note uses an Instagram-like action to demonstrate backend necessity. A simple tap on a like button actually triggers multiple coordinated backend operations.

  • The app sends the like event to the server.
  • The server authenticates the user.
  • The like is persisted in a database.
  • The post owner is identified and a notification is sent.

Centralized data and state management

The note emphasizes that each user experience is personalized according to profile, actions, and relationships. That requires a centralized system that can maintain state and route the right interactions to the right users in real time.

Core backend responsibility: data management

The note reduces the backend to its essential data role: fetching data, receiving data, persisting data, and performing actions on data. That data-centric role is presented as the heart of backend engineering.

Chapter 4

How frontend runtime differs from backend runtime

The source note then asks a natural question: if frontend devices are also computers, why not perform backend work there? To answer that, it explains how frontend code actually runs and what the browser allows it to do.

Frontend application flow in a Next.js example

In the noteâ€™s Next.js example, the browser first receives an HTML document and then separately fetches JavaScript, CSS, fonts, and images. Those assets are what let the browser render and hydrate the application.

  • The frontend domain is routed through NGINX to localhost port 3000.
  • The browser becomes the runtime that executes the received frontend code.
  • Hydration attaches event listeners and enables interactivity on the client.

The essential runtime difference

The backend runs on the server machine with access to the operating system and runtime environment. The frontend runs inside the userâ€™s browser, which is a sandboxed and restricted execution environment.

Aspect Backend Server Frontend Browser
Execution location Server machine such as an EC2 instance User device inside the browser
Processing model Processes requests and returns results Executes received JavaScript and renders UI
System access Broad access to runtime and OS resources Sandboxed with restricted resource access

Browser sandboxing and CORS

The browser security model is one of the main reasons frontend code cannot replace the backend. The source note describes the browser as a sandbox that can access only browser APIs such as the DOM, cookies, and local storage.

  • Browser code does not have unrestricted access to file systems or native OS resources.
  • Cross-origin calls are restricted by CORS unless the target server explicitly allows them.
  • These restrictions exist to prevent malicious remote code from abusing the userâ€™s device or data.

Chapter 5

Why backend logic must remain server-side

The final part of the source note turns the comparison into a conclusion. Backend logic remains server-side because the browser lacks the security permissions, connectivity model, and system resources needed for full backend behavior.

Key reasons frontend cannot fully replace the backend

  • Browsers cannot access environment variables, file systems, or server-side configuration in the same way a backend can.
  • Frontend code is limited by external API restrictions such as CORS.
  • Browsers cannot maintain persistent database connections or raw socket communication like native database drivers.
  • Connection pooling is a server-side scalability technique that browsers cannot realistically provide.
  • User devices have inconsistent CPU and memory capacity, making centralized backend execution more reliable for heavy logic.

Database communication and scaling constraints

The source note singles out database access as a major dividing line. Backend servers can use native database drivers with persistent connections, binary protocols, and connection pools. Browsers cannot do that safely or efficiently at scale.

Conclusion and key takeaways

The note ends by restating the backend as a centralized server-side system responsible for data, state, and business logic. The frontend remains focused on rendering and client-side interaction, while the backend carries the work that depends on security, connectivity, persistence, and scale.

  • DNS, firewalls, and reverse proxies are part of the real backend path, not side details.
  • Browser sandboxing explains why frontend logic is fundamentally constrained.
  • Backend architecture remains necessary for scalability, performance, security, and persistent state.
